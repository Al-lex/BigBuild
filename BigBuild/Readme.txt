1. Установить приреквизиты  в следующем порядке:
 1. python-3.4.2.msi -обязательно выбрать пункт прописывающий python в переменные среды
 2.pymssql-2.1.1.win-amd64-py3.4.exe (если не будет ставиться то pymssql-2.1.1.win32-py3.4)
 3.sqlncli.msi
 4.SqlCmdLnUtils.msi
2. В файле MW.config прописать настройки -каждая branch соответствует ветке разработки, вне branch хранятся общие настройки для всех веток. Внимание! Плагины берутся общие для всех веток 
3.Запустить PS от администратора и выполнить команду Set-ExecutionPolicy Unrestricted
4.Перейти в каталог со скриптом BigBuild.py и запустить его
5.Лог выполнения пишется в UpdateLog.log
6.Лог sql скриптов пишется в resultrestor.txt в папке с названием ветки


BigBuild.py поддреживает параметризованный запуск. Для запуска не всего решения. а только определенной части нужно указать сооответствующий параметр:
1  -запуск только обновления БД
2  -копирование файлов основного приложения, копирование и установка службы платежей и редактирование web.config основного приложения
3  -установка плагинов
4  -копирование вспомагательных файлов (MT-Data и т.п.)
5  -установка основного приложения на IIS
6  -установка веб-сервисов на IIS. редактирование конфигов веб-сервисов

Параметры передаются через пробел, например BigBuild.py 4 5  - выполнение 4 и 5 шагов

Формат файла MW.config
Внимание! БД должна быть в рамках текущего релиза (номер сервиспака не важен). 
Основная логика содержится в секции branch - она определяет ветку которая будет собираться. Теоретически можно собирать несколько бренчей одновременно (не тестировалось!).
pathToLatest - путь к файлам ветки
pathToLocal - путь к локальной директории где будут расположены файлы приложения
iisAppSettings - под каким именем и в какой пул ставить приложение
isLastBuild - пока не работает (не трогаем)
Services - список сервисов которые ставим. Чтобы сервис ставился надо указать true

Plugins - плагины которые ставим (указывать название точно совпадающее с названием в сборке). В текущей версии плагины одинаковы для всех веток. Почему так сделано? Изначально так задумали, пока что не переделал
Files -дополнительные файлы (должны быть в папке Extra)
Directories -статические директории - временная подпорка для всякой кривоты которая еще не вошла в билдпроцесс


dbconnection - описание настроек БД, включая путь к бекапу. Для того чтобы не происходило восстановления из бекапа нужно isRestore установить false
